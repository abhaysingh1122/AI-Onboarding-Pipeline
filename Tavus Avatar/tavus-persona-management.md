## API Field Names (CRITICAL)
- **Objectives**: `objective_name`, `objective_prompt` — correct
- **Guardrails**: `guardrail_name`, `guardrail_prompt` — SINGULAR, NOT plural
- **Wrong**: `guardrails_name`, `guardrails_prompt` → API returns 400 "Unknown field"
- The old `create_persona.sh` used plural names, which is why guardrails silently failed to create. Discovered 2026-03-11.

## Updating an Existing Persona (PATCH)
- **Endpoint**: `PATCH /v2/personas/{persona_id}`
- **Content-Type**: `application/json-patch+json`
- **Format**: RFC 6902 JSON Patch — array of `{ op, path, value }` objects
- **Updatable fields**: `system_prompt`, `persona_name`, `objectives_id`, `guardrails_id`, `document_ids`, all layer configs
- **Example**: `[{ "op": "replace", "path": "/system_prompt", "value": "new prompt" }]`

## Best Practice: Update Flow
Don't PATCH objectives/guardrails data arrays directly — complex and error-prone. Instead:
1. Edit config files locally (`system_prompt.txt`, `objectives.json`, `guardrails.json`)
2. Run `node update_persona.mjs` — POSTs new objectives+guardrails, PATCHes persona
3. If KB changed: edit `knowledge_base.txt`, then run `node upload_kb.mjs`
4. If `conversational_context` changed: edit `useTavusAgent.ts`, redeploy app via Railway
5. Always verify with GET persona after deploy — check `objectives_id` and `guardrails_id` are not null

Old objectives/guardrails resources remain in the Tavus account but are detached.

## Content Architecture — What Goes Where
Zero duplication between layers. Each layer has one job:

| Layer | Owns | Does NOT contain | Config file |
|---|---|---|---|
| System prompt | Personality, tone, turn format, greeting command, skip-ahead | Behavioral rules, question content, reference data | `system_prompt.txt` |
| Objectives | Conversation flow, branching, questions, tool sequencing | Personality, behavioral constraints | `objectives.json` |
| Guardrails | Hard behavioral boundaries | Personality, question flow | `guardrails.json` |
| Knowledge Base | RAG reference data (company info, tone examples) | Instructions, question lists, behavioral rules | `knowledge_base.txt` |
| conversational_context | Per-session directive (normal vs restart) | Personality (system prompt owns that) | `useTavusAgent.ts:370-373` |

## Tool Registration
- Tools live in `layers.llm.tools` on the persona (registered via dashboard or API)
- **Do NOT mention tools in system_prompt** — wastes tokens, no benefit
- Objectives tell the LLM WHEN to call a tool (closing objective in `objectives.json`: "call the end_session tool")
- Tavus auto-injects tool schemas into the LLM context at runtime
- Local reference copy: `end_session_tool.json`

## Knowledge Base Upload
- **API requires a URL** (`document_url` field) — cannot upload inline content
- **Workaround**: `upload_kb.mjs` creates a GitHub Gist from `knowledge_base.txt`, POSTs the raw URL to Tavus, attaches to persona, deletes the gist, and auto-updates `persona_ids.txt`
- **Self-syncing**: Script reads `KB_DOCUMENT_ID` from `persona_ids.txt` and writes the new ID back after upload — no manual ID management needed
- **Prerequisite**: `gh` CLI must be authenticated (`gh auth status`)
- **Processing**: Status goes `started` → `ready` (takes a few minutes)
- **Recrawl**: `POST /v2/documents/{id}/recrawl` — 1-hour cooldown, only for URL-based docs

## Greeting Fix
- Two fields in the conversation creation body (`useTavusAgent.ts`) trigger greeting:
  - `custom_greeting` — platform-level: Tavus speaks this text immediately when the user joins, no LLM latency
  - `conversational_context` — LLM-level: imperative instruction ("Speak first — greet the client immediately. Do not wait.")
- `custom_greeting` is the primary mechanism; `conversational_context` is a reinforcing backup
- System prompt greeting should also be imperative (first instruction in `system_prompt.txt`, not buried)
- The `greeting` field on the persona exists but is optional — `custom_greeting` + `conversational_context` + system prompt covers it

## Token Optimization
- Every token affects latency — shorter prompts = faster TTFT regardless of model
- Biggest win: remove cross-layer duplication (system prompt restating guardrails)
- Session result: 2,143 → 998 chars system prompt (53%), ~3,284 → ~1,870 total tokens (43%)
- Use concrete speech examples ("love that", "oh nice") instead of abstract adjectives ("warm, friendly")
- Keep `system_prompt.txt` under ~1,000 chars — it's loaded every LLM turn
- **Token limits** (from Tavus docs): 5K optimal, 15-20K performance degrades, 32K hard maximum
- See `ressources/REFERENCE - Tavus Persona CVI Overview.md` for full layer config reference (LLM models, TTS engines, Sparrow stats, perception options)

## Tavus API Quirks
- **PATCH 304**: Tavus returns HTTP 304 (Not Modified) when a JSON Patch results in no actual change. This is success, not an error — the value already matches. Empty response body.
- **Intermittent 401s**: Server-side issue. Retry usually works. Not an API key problem.
- **`objectives_id: null`**: Means objectives were never successfully attached — verify after deploy
- **`guardrails_id: null`**: Same — always verify with GET after PATCH
- **`document_ids`**: Array of document IDs. PATCH with `op: "replace"` to swap docs.
